# Управление дырой (виртуальный «стик» от клика)

Реализация: [`src/core/gameState.js`](../src/core/gameState.js). Ввод подключается из [`src/app/bootstrap.js`](../src/app/bootstrap.js) через `attachPointerDrag`.

## Модель

После **нажатия ЛКМ** фиксируется **центр управления** — точка клика в нормализованных координатах контейнера `#game-container`:

- `nx, ny ∈ [0, 1]` по ширине и высоте контейнера (левый верх — `(0, 0)`).

Пока кнопка зажата, положение курсора задаёт **отклонение стика**:

\[
\vec{s} = (pointer - controlCenter)
\]

- **Направление** движения дыры совпадает с направлением отклонения **в пикселях** \((s_x \cdot v_w,\, s_y \cdot v_h)\), где \(v_w, v_h\) — `layout.designWidth/Height` (изотропный круг «стика» на экране).
- **Модуль скорости** зависит только от **длины** этого пиксельного отклонения, а не от того, движется ли мышь в данный кадр: если курсор **стоит** на том же смещении от клика, дыра **продолжает** двигаться с той же скоростью.

Дыра **не** притягивается к экранной позиции курсора как к цели на поле; используется только вектор «клик → курсор».

## Формула скорости

Пусть `worldW = designWidth * m`, `worldH = designHeight * m`, `m = WORLD_MAP_VIEW_MULTIPLIER`, `minSpan = min(worldW, worldH)`, `minCss = min(designWidth, designHeight)`.

1. Пиксельное смещение стика: `(dx, dy) = (s_x * designWidth, s_y * designHeight)`, `distPx = hypot(dx, dy)`.
2. Если `distPx < HOLE_STICK_DEAD_ZONE * minCss` — нейтраль, скорость **0**.
3. Иначе:
   - направление на экране: `(ux, uy) = (dx, dy) / distPx`;
   - `tilt = min(1, distPx / (HOLE_STICK_RANGE * minCss))`;
   - `pixPerSec = HOLE_MAX_SPEED * minSpan * tilt` — одинаковая скорость сдвига поля в px/s по осям при полном наклоне;
   - скорость на карте: `holeVnX = ux * pixPerSec / worldW`, `holeVnY = uy * pixPerSec / worldH` (согласовано с `setScroll` в `createPlayfield.js`).

Интеграция позиции на **карте**: `map += holeVn * dt` (с ограничением `dt` в коде).

## Константы

| Константа | Смысл |
|-----------|--------|
| `HOLE_MAX_SPEED` | Множитель: при полном наклоне сдвиг поля в px/s по экрану не больше `HOLE_MAX_SPEED * min(worldW, worldH)`. |
| `HOLE_STICK_RANGE` | Множитель к `min(designWidth, designHeight)`: пиксельное смещение до насыщения скорости. |
| `HOLE_STICK_DEAD_ZONE` | Множитель к `min(designWidth, designHeight)`: радиус мёртвой зоны в px. |
| `HOLE_RELEASE_FRICTION` | После отпускания ЛКМ — экспоненциальное затухание скорости (~за секунду). |
| `WORLD_MAP_VIEW_MULTIPLIER` | Размер карты: на сколько раз больше вьюпорта по каждой оси ([`constants.js`](../src/core/constants.js)). Границы `mapN` согласованы с ним, чтобы вью оставался в зоне поля. |
| `getMapPositionBounds01()` | Диапазон \([1/(2m), 1-1/(2m)]\) для `mapNx/Y`. |

Подбирайте `HOLE_STICK_RANGE` и `HOLE_MAX_SPEED` вместе: при малом диапазоне дыра быстрее выходит на максимум при коротком движении мыши.

## Состояние (`GameState`)

- `controlCenterNx/Y` — точка клика (центр стика).
- `pointerTargetNx/Y` — текущие нормализованные координаты указателя (обновляются в `pointermove`).
- `mapNx/Y` — позиция дыры на **карте** в норм. координатах `[0, 1]` (логическое поле больше экрана в `WORLD_MAP_VIEW_MULTIPLIER` раз).
- `holeVnX/Y` — текущая скорость дыры (при отпускании используется для инерции с трением).

## Жизненный цикл ввода

1. **`beginPointerDrag(state, nx, ny)`** — `pointerdown`: записать центр, сбросить скорость.
2. **`setPointerTarget(state, nx, ny)`** — `pointermove` при захвате: обновить курсор.
3. **`stepHolePhysics(state, dt, layout)`** — каждый кадр: пересчитать скорость и позицию (нужен актуальный layout для изотропии в px).
4. **`endPointerDrag(state)`** — `pointerup`: выключить режим стика; скорость гасится по `HOLE_RELEASE_FRICTION`.

## Границы поля

Чтобы **камера (центр экрана) не уезжала за пределы нарисованного полотна** при `WORLD_MAP_VIEW_MULTIPLIER = m`, норм. координаты позиции на карте ограничиваются «внутренней» полосой \([1/(2m),\,1-1/(2m)]\) по каждой оси (см. `getMapPositionBounds01()` в [`constants.js`](../src/core/constants.js)). При упоре в границу соответствующая компонента скорости обнуляется.

## Визуальный джойстик

Пока `state.dragging === true`, поверх игры показывается оверлей ([`src/ui/holeJoystick.js`](../src/ui/holeJoystick.js)):

- **База** — полупрозрачный круг с обводкой; центр в `(controlCenterNx, controlCenterNy)` в долях ширины/высоты контейнера.
- **Ручка** — меньший круг (~38% диаметра базы); центр ручки всегда в позиции курсора. Пока расстояние «центр базы (точка клика) → курсор» невелико, база неподвижна. Если ручка **вылезает** за окружность базы больше чем на **3/4 своего радиуса** (вдоль луча к курсору), **база сдвигается** к курсору так, что выступ ровно **3/4 радиуса ручки**; логика игры по-прежнему от фиксированного `controlCenter` в `gameState`.
- **Резкий разворот:** если за кадр вектор движения курсора (в пикселях) по отношению к предыдущему шагу разворачивается достаточно резко (`dot` ниже порога в `holeJoystick.js`), включается **заморозка базы**: внешний круг не двигается, внутренний идёт за курсором. Снятие заморозки — когда курсор снова в пределах `maxKnobDist` от **текущей** нарисованной базы; тогда база **согласуется** с идеальной геометрией (клик + курсор).
- Слой с `pointer-events: none`, `z-index: 2` (выше Three.js-канваса).

## Связь с рендером

- **Ввод** по-прежнему в **норм. координатах контейнера** (как в `pointerDrag`).
- **Карта и камера**: логическая позиция — `mapNx/Y`. Слой Pixi ([`createPlayfield.js`](../src/render/pixi/createPlayfield.js)) **смещает полотно** так, что точка карты `(mapNx, mapNy)` оказывается в центре вьюпорта: «камера следует за дырой», дыра на экране не уезжает к краям вместе с миром.
- **Дыра (Three.js)** рисуется **по центру** вью ([`createHoleView.js`](../src/render/three/createHoleView.js), `setScreenCentered`); с миром синхронизирует только Pixi.
- Размеры layout: [`src/core/viewport.js`](../src/core/viewport.js). Для произвольного размещения дыры в экранных координатах по-прежнему можно использовать `containerToDesignNormalized` (см. `setPointerNorm` в `createHoleView`).
